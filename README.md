# wedding-invitation-landing-page

# 結婚式招待状ランディングページ 要件定義書 (AWS版)

## 1. 概要
本ドキュメントは、新郎新婦の結婚式招待状となるWebランディングページ（以下、LP）を作成するにあたり、その仕様と要件を定義するものである。GitHub Copilot等を用いた開発のインプットとして、またAWS上での構築を前提とした技術仕様書として使用することを目的とする。

## 2. 画面・コンテンツ要件
LPに表示するコンテンツと、その具体的な内容を定義する。

### 2.1. 必須要件
招待状として最低限必要な情報を定義する。

| No. | 要件名 | 具体的な指示（内容） |
| :-- | :--- | :--- |
| 1 | **メインビジュアル** | - ページの最上部に、画面幅いっぱいに表示されるヒーローイメージを配置する。<br>- 画像は指定された新郎新婦の写真を使用する。<br>- 画像の上に「〇〇 & △△'s Wedding Party」といったタイトルと結婚式の日付を重ねて表示する。 |
| 2 | **挨拶文** | - メインビジュアルの下に、ゲストへの感謝と結婚報告、招待の旨を記載した挨拶文を配置する。<br>- テキスト内容は別途指定されたものを使用する。 |
| 3 | **新郎新婦の名前** | - 挨拶文の下、または見出しとして新郎新婦のフルネームを記載する。 |
| 4 | **結婚式・披露宴の日時と場所** | - 以下の情報をリスト形式などで分かりやすく表示する。<br>  - **日時**: 202X年XX月XX日 (土)<br>  - **受付**: HH:MM～<br>  - **挙式**: HH:MM～<br>  - **披露宴**: HH:MM～<br>  - **場所**: 〇〇迎賓館 △△の間 |
| 5 | **出欠回答フォーム** | - ゲストが出欠席を登録するためのフォームを設置する。<br>- **入力項目**: <br>  - お名前 (必須)<br>  - ふりがな (必須)<br>  - ご出席/ご欠席 (必須・ラジオボタン)<br>  - メールアドレス (必須)<br>  - アレルギーの有無 (テキストエリア)<br>  - メッセージ (テキストエリア)<br>- 「送信」ボタンを配置し、後述するAWSのバックエンドと連携するフォーム送信機能を実装する。<br>- 必須項目には入力バリデーション（未入力チェック等）を実装する。 |
| 6 | **回答締切日** | - 出欠回答フォームの上部、または下部に回答締切日を明記する。<br>- 例: 「誠に恐れ入りますが 202X年XX月XX日までにご返信くださいますようお願い申し上げます」 |

### 2.2. 推奨要件
ゲストの利便性向上や、より丁寧な案内のために実装が推奨される情報。

| No. | 要件名 | 具体的な指示（内容） |
| :-- | :--- | :--- |
| 7 | **会場へのアクセス情報** | - 会場の正式名称、住所、電話番号を記載する。<br>- Googleマップを埋め込み(`iframe`)、地図上で場所を確認できるようにする。<br>- 最寄り駅からの徒歩時間や、主要駅からのアクセス方法をテキストで補足する。 |
| 8 | **当日のタイムライン** | - 「受付」「挙式」「披露宴」「お開き」などの当日の大まかな流れと予定時刻をリスト形式で表示する。 |
| 9 | **ドレスコードの案内** | - 服装について指定がある場合（例: 「平服でお越しください」）、その旨を記載するセクションを設ける。 |
| 10 | **会費・ご祝儀に関する案内** | - 会費制の場合は、金額と受付での支払い方法を明記する。<br>- ご祝儀を辞退する場合は、その旨を丁寧に記載する。 |

### 2.3. 任意要件
二人らしさを演出し、ゲストの満足度を高めるための追加情報。

| No. | 要件名 | 具体的な指示（内容） |
| :-- | :--- | :--- |
| 11 | **二人のプロフィール** | - 新郎・新婦それぞれのセクションを設け、顔写真（またはイラスト）と簡単な自己紹介（趣味、馴れ初めなど）を掲載する。 |
| 12 | **写真ギャラリー** | - 前撮り写真などを複数枚表示できるギャラリーを実装する。<br>- 画像をクリックすると拡大表示されるライトボックス機能を実装することが望ましい。 |
| 13 | **Q&Aセクション** | - 「駐車場はありますか？」など、ゲストから想定される質問と回答をまとめたQ&Aセクションを設ける。<br>- クリックで回答が開閉するアコーディオンUIで実装する。 |
| 14 | **パスワード保護** | - 招待状を受け取ったゲストのみが閲覧できるよう、ページに簡易的なパスワード保護をかける。<br>- ページアクセス時にパスワード入力を求めるモーダルウィンドウを表示する。 |

## 3. 機能要件
本LPが提供すべきシステムとしての機能を定義する。

| 機能分類 | 機能名 | 機能概要と具体的な実装内容 |
| :--- | :--- | :--- |
| **出欠管理** | **出欠回答フォーム機能** | ゲストが出欠情報を入力し、新郎新婦宛に送信できる。<br>- **入力値検証**: 必須項目の未入力、メールアドレス形式のクライアントサイド・バリデーションを実装する。<br>- **データ送信**: 「送信」ボタンクリックで、フォームの入力内容を後述のAWSバックエンド（API Gateway）に非同期で送信する。<br>- **送信完了処理**: 送信完了後、「ご回答ありがとうございます」などの完了メッセージを画面に表示する。 |
| | **出欠データ通知機能** | フォームから送信されたゲストの情報を、AWS LambdaとSESを用いて処理し、指定されたメールアドレスに通知する。これにより、新郎新婦はリアルタイムに回答を確認できる。 |
| **情報表示** | **会場マップ表示機能** | Google Maps PlatformのAPIを利用し、指定された住所の地図をページ内に埋め込む (`iframe` タグ)。地図上には会場を示すピンを立てる。 |
| | **写真ギャラリー機能** | 複数のサムネイル画像を表示し、クリックすると拡大表示されるライトボックスUIを実装する。拡大表示中にナビゲーション（次へ/前へ）や閉じるボタンを実装する。 |
| **UI/UX** | **レスポンシブデザイン対応** | CSSのメディアクエリを用いて、PC、タブレット、スマートフォンなど異なる画面幅に応じた最適なレイアウトを提供する。モバイルファーストな設計を基本とする。 |

## 4. 非機能要件
システムの品質、性能、制約など、機能以外の要件を定義する。

| 分類 | 要件名 | 具体的な内容 |
| :--- | :--- | :--- |
| **パフォーマンス** | **ページ表示速度** | ゲストがストレスなく閲覧できるよう、ページの読み込みを高速化する。<br>- 画像ファイルは圧縮し、WebPなどの次世代フォーマットを利用してファイルサイズを削減する。<br>- JavaScriptの遅延読み込み（lazy loading）を実装し、初期表示速度を向上させる。 |
| **互換性** | **クロスブラウザ対応** | 主要なWebブラウザ（Google Chrome, Apple Safari, Mozilla Firefox, Microsoft Edge）の最新バージョンで表示や機能が正常に動作すること。 |
| **ユーザビリティ** | **直感的な操作性** | ITリテラシーに関わらず、全てのゲストが迷わずに出欠登録を完了できること。<br>- フォームの入力項目には分かりやすいラベルや入力例（プレースホルダー）を示す。<br>- ボタンやリンクは、クリック可能であることが視覚的に分かるデザインにする。 |
| **アクセシビリティ** | **基本的なWebアクセシビリティ** | 高齢者や障害を持つゲストにも配慮し、画像には代替テキスト（alt属性）を設定し、テキストと背景には十分なコントラスト比を確保する。 |
| **保守・運用性** | **情報の更新容易性** | 将来的に挨拶文や写真などを差し替える際に、専門知識がなくても容易に更新できるよう、テキスト情報や画像パスなどを設定ファイル（例: JSON）に分離して管理する。 |

## 5. システム構成（AWS）

### 5.1. フロントエンド構成
静的Webページであるため、サーバーレスアーキテクチャを採用し、コスト効率とメンテナンス性、スケーラビリティを確保する。

- **概要図**:
  ```mermaid
  graph TD
      A[ゲスト] -->|HTTPS| B(Amazon CloudFront);
      B -->|コンテンツ取得| C(Amazon S3);
      subgraph "AWS"
          B;
          C[S3バケット<br>HTML/CSS/JS/画像];
      end
  ```
- **主要サービス**:
  - **ストレージ (Amazon S3)**: ビルドされた静的ファイル（HTML, CSS, JS, 画像）を保管。
  - **コンテンツ配信 (Amazon CloudFront)**: S3上のコンテンツを高速配信するCDN。
  - **SSL/TLS証明書 (AWS Certificate Manager)**: CloudFrontと連携し、無料でHTTPS通信を有効化。
  - **ドメイン管理 (Amazon Route 53)**: (任意) カスタムドメインを利用する場合に使用。

### 5.2. バックエンド構成（出欠フォーム処理）
フォームから送信されたデータを処理するために、APIをサーバーレスで構築する。

- **概要図**:
  ```mermaid
  graph TD
      subgraph "ゲストのブラウザ"
          D[出欠回答フォーム]
      end
      
      subgraph "AWS"
          E(API Gateway) -->|リクエスト転送| F(AWS Lambda);
          F -->|メール送信指示| G(Amazon SES);
      end

      D -- "フォーム送信 (POSTリクエスト)" --> E;
      G -- "出欠情報をメールで通知" --> H((新郎新婦));
  ```
- **主要サービス**:
  - **APIエンドポイント (API Gateway)**: ブラウザからのフォーム送信リクエストを受け付ける窓口。
  - **データ処理 (AWS Lambda)**: 受け取ったフォームデータを処理し、メール送信用に整形するプログラム（Python, Node.js等）を実行。
  - **メール送信 (Amazon Simple Email Service - SES)**: Lambdaからの指示に基づき、整形された出欠情報を指定のメールアドレス宛に送信。
